## Decisions

### DTMF grammar correction
The `LoadConfig` command was originally coded as `#N**` (two stars) to match
what had been described in the session. It was later corrected to `#N*` (one
star) to match the actual C# implementation behavior. The final grammar in
`src/rumble/dtmf.py` is:
- `#*` — Disconnect
- `#N*` — LoadConfig(bank=N)
- `#XX#Y*` — AdminSetting(setting, value)
- `#XXX#Y*` — ChangeChannel(server_number, channel_number)

### pymumble SSL shim
pymumble 1.6.1 calls `ssl.wrap_socket()`, which Python 3.12 removed. Rather
than downgrade Python or wait for a pymumble release, a shim was installed at
the top of `src/rumble/mumble_client.py` that re-implements `wrap_socket` via
`ssl.SSLContext`. The shim is guarded with `if not hasattr(ssl, "wrap_socket")`
so it is a no-op on Python 3.11. `pymumble` is pinned to `==1.6.1` in
`pyproject.toml` so a new release cannot silently change SSL behavior.
CLAUDE.md "Known workarounds" has the full removal procedure for when upstream
ships a fix.

### Docker dev server config approach
The `mumblevoip/mumble-server` Docker image silently ignores a bind-mounted
`.ini` file. Configuration is applied via `MUMBLE_CONFIG_*` environment
variables instead. The file `docker/mumble-server.ini` was deleted. See
`docker/docker-compose.yml` for the current env var list.

### Web UI security stance
CSRF protection is deliberately absent from the web UI POST endpoints. The
current config (`web.host: 127.0.0.1`) is loopback-only by default, which is
the accepted mitigation for now. The issue is tracked in
`docs/issues-to-file.md`. Do not expose on `0.0.0.0` on untrusted networks
without adding CSRF tokens first.

### Web UI error banners
The "reload config" endpoint returns an HTML fragment (HTMX swap target) rather
than JSON, so error messages appear inline in the page. HTML-escaping is applied
to config error text before insertion into the DOM.

### Documentation register
The user-facing docs are written for a licensed amateur radio operator who may
not know VOIP. Voice is friendly-expert, active, uses "operator" not "user",
references FCC §97.119, uses callsign AE9S for the node and fictional callsigns
(W0XYZ, N1ABC, K9TST) for examples. Full conventions are in `CLAUDE.md` under
"Documentation conventions."

### C# comparison removed from README
The "Comparison to the original C# Rumble" section was removed from README.md
because the author made the original C# repository private. A brief mention of
the project history remains in the author bio line. No other doc links to the
now-private repo URL.

## Pending / next session

These are the remaining milestones. Order matters: audio passthrough first,
then ident, then PTT, then hardening.

1. **Milestone 7a — radio audio passthrough to Mumble.** The `AudioCapture`
   worker in `src/rumble/audio.py` currently only feeds the `DtmfDetector`.
   The PCM samples also need to be forwarded to `MumbleClient.send_audio()`
   when the node is connected. The hook point is the audio worker thread in
   `AudioCapture`; add an optional second callback (or extend the existing one)
   that receives raw PCM and calls `self._mumble.send_audio(chunk)`. This is
   the single biggest missing feature.

2. **Milestone 7b — incoming Mumble audio to radio speaker.** `MumbleClient`
   has `on_audio_received()` (see line 458 of `src/rumble/mumble_client.py`)
   which registers a callback for incoming `MumbleAudioFrame` objects. Wire
   that callback in the dispatcher to push frames to `sounddevice` output —
   this requires a PTT keying step before transmitting to the radio.

3. **Milestone 7c — implement `src/rumble/ident.py`.** Read `IdentConfig`
   (fields: `callsign`, `interval_minutes`, `wav_file`, `enabled`) from the
   dispatcher at startup. Fire a `threading.Timer` loop; when it fires, play
   the WAV via `MumbleClient.send_audio()` or synthesize CW via TTS. The
   config fields already exist and are validated by the config loader.

4. **Hardware PTT.** No PTT keying is implemented anywhere. Three options
   (any or all can be offered via config): CAT serial commands, GPIO pin
   (Raspberry Pi), serial DTR/RTS. Config field for PTT method should live
   under `audio:` in the YAML. No code exists yet.

5. **CSRF on web UI POST endpoints.** See `docs/issues-to-file.md`.
   Do not ship `web.host: 0.0.0.0` as a documented option without this.

6. **Packaging.** README says "one-line install on a Raspberry Pi" is coming.
   `pyproject.toml` has the package structure; no wheel/sdist build pipeline
   or PyPI release mechanism exists yet.

## Open questions

- **PTT design (for the user to decide).** Of the three PTT options (CAT,
  GPIO, serial DTR/RTS), does AE9S want all three configurable from YAML, or
  just serial DTR/RTS first since it's the most common sound-card interface
  wiring? This affects how much abstraction to build into the PTT layer.

- **Incoming audio resampling.** pymumble delivers audio at 48 kHz internally
  (`PYMUMBLE_SAMPLERATE = 48000`). The sounddevice output device may be
  configured at a different sample rate. No resampling logic exists yet —
  will need `numpy` or `scipy.signal.resample` in the audio-out path.

- **CW ident format.** `IdentConfig.wav_file` is a path to a pre-made WAV.
  Is that the preferred approach, or does AE9S want runtime CW synthesis?
  The `tts.py` approach via pyttsx3 reads the callsign as speech, not CW.

## Watch out for

### Python version
The machine runs Python 3.12.3. `pyproject.toml` says `>=3.11`. The SSL shim
in `src/rumble/mumble_client.py` (lines ~36-60) is required for 3.12+. Do not
remove it until pymumble ships a fixed release and you have re-run the full
integration suite (see CLAUDE.md "Known workarounds" for the exact steps).

### `src/rumble/ident.py` is a stub
The file looks like a module but contains only a single comment line. Anything
that tries to import a class or function from it will fail with `ImportError`.
Do not import from it until it is implemented.

### Integration tests require Docker
`tests/` contains integration tests guarded by `RUMBLE_INTEGRATION=1`. Run
them with:
```
cd /home/kb/Desktop/Projects/rumble-py/docker && docker compose up -d
cd /home/kb/Desktop/Projects/rumble-py
RUMBLE_INTEGRATION=1 pytest -v -k mumble
```
Without `RUMBLE_INTEGRATION=1`, those tests are skipped, not failed — so a
clean `pytest` run does not confirm integration health.

### HTMX SRI hash
The HTMX script tag in the web UI templates uses a hardcoded SRI (Subresource
Integrity) hash pinned to a specific HTMX version. If HTMX is ever upgraded,
the hash must be updated or the browser will refuse to load the script.
